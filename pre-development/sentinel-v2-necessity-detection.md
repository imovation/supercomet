# 哨兵机制 v2：必要性检测

## 当前哨兵 v1

`ci-sentinel.yml` 每日安装 `@rpamis/comet@latest` 并运行 supercomet 的全部 BATS 测试。它只回答一个问题：

**supercomet 的增强在最新 Comet 上还能用吗？**

## v1 的盲区

它不回答另一个问题：

**这些增强还有必要存在吗？**

| 场景 | v1 行为 | 应发生 |
|------|---------|--------|
| 上游 Comet 原生实现了 bidirectional-verify | 测试可能仍通过（不冲突） | 检测到并通知用户：上游已内置，可移除 supercomet 版本 |
| 上游的实现比 supercomet 更好 | 静默 | 检测差异，标注"上游推荐" |
| 上游的实现比 supercomet 差 | 静默 | 检测差异，标注"保持 supercomet 增强" |
| 上游新增了与 supercomet 冲突的功能 | 测试失败 | 区分"冲突"还是"过时"，给出修复建议 |
| 上游无变化 | 通过 | 不提示 |

**核心问题**：哨兵只检测可行性（能不能用），不检测必要性（还需不需要）。

## 建议方案：三层哨兵体系

```
┌─────────────────────────────────────────────┐
│              哨兵 v2 架构                    │
├─────────────────────────────────────────────┤
│                                             │
│  Layer 1: 兼容性（v1，保持）                │
│  ├─ 安装 @rpamis/comet@latest               │
│  ├─ 运行 BATS 测试                          │
│  └─ 失败 → 告警                             │
│                                             │
│  Layer 2: 功能重合检测（新增）              │
│  ├─ 对比 dist/version.yaml 声明的增强列表   │
│  ├─ 与上游 Comet changelog/spec 对比         │
│  ├─ 检测上游是否已实现相同功能               │
│  └─ 输出重合矩阵：谁实现了什么、谁更好       │
│                                             │
│  Layer 3: 决策建议（新增）                  │
│  ├─ 上游实现更好 → 建议废弃                  │
│  ├─ 上游实现更差 → 建议保持                  │
│  ├─ 上游未实现 → 建议向上游提 PR              │
│  └─ 冲突 → 建议适配                          │
│                                             │
└─────────────────────────────────────────────┘
```

## 实现要点

- `dist/manifest.yaml` 声明每个增强的签名（脚本名、功能特征、关键输出）
- CI 对比上游 Comet 的 changelog/spec 文件
- 无自动化判断"谁更好"——输出对比报告，让人决策
- 每日运行，结果输出为 `sentinel-report.md` 提交到仓库
