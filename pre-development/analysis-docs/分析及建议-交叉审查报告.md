## 三、冲突裁决（3项）

### 冲突1：Triple Arbitration 状态恢复

| | 本会话 | 分析及建议 | 裁决 |
|---|--------|----------|------|
| **结论** | 必须吸收（A2，高优先级） | 明确不迁移 | **本会话高估了价值** |

**裁决**：不独立成一个吸收项。将 Triple Arbitration 降级为轻量一致性校验——在 `comet-state.sh` 的 `check` 子命令中加入：当 `.comet.yaml` 的 phase 与 git log 中的任务完成状态明显矛盾时，输出警告而非静默信任。不改变 Comet 的状态机架构。优先级降至**低**。

### 冲突2：7 项验证清单

| | 本会话 | 分析及建议 | 裁决 |
|---|--------|----------|------|
| **结论** | 吸收但重构为 5 项（A4，高优先级） | 太重，只取双向反查和 revert-restore | **分析及建议的"解耦+选择性吸收"更务实** |

**裁决**：采纳分析及建议的方案。双向反查单独吸收（P0），revert-restore 作为可选脚本（P2），i18n gate 不入通用方案。

### 冲突3：spec 中途变更的处理

**裁决**：分析及建议的判断正确。Comet 的"推迟裁决"比 specpower 的"级联回退"更工程化。不吸收级联回退矩阵。
